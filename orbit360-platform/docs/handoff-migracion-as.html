1 · Crear el nuevo proyecto A&S

Orbit 360 es la versión comercializable (genérica, datos ficticios). A&S es el primer cliente: se personaliza SOLO por configuración, sin tocar el core.

Tipo de proyecto

Proyecto normal de Claude (no Design). Design es solo para prototipos visuales; A&S es trabajo de producto continuo sobre código real.

1
Crear el proyecto
En claude.ai → "New Project" → nombre: "Orbit 360 · Alianzas y Soluciones".
2
Subir el ZIP base
Sube el ZIP de orbit360-platform v1.54 (o la última versión). Esa carpeta ES el repo completo.
3
Pegar las instrucciones del proyecto
Copia el contenido del CLAUDE.md que está en la raíz del ZIP en "Project Instructions". Ahí están las reglas fijas, arquitectura y forma de continuar.
4
Primer mensaje
"Configura Orbit 360 para A&S. Lee CLAUDE.md y docs/PLAN-INFRAESTRUCTURA.md. Personaliza solo por config: Orbit.tenant, seed de aseguradoras reales, asesores, metas."
⚠️ Regla de oro: NUNCA modifiques el core para personalizar A&S. Toda personalización va en core/config.js (Orbit.tenant) y data/seed.js. Así puedes actualizar el core sin romper la personalización.

2 · Conectar el backend

Hoy la plataforma corre 100% en el navegador con localStorage. Para producción real (multi-usuario, datos persistentes en la nube) se swappea la capa de datos.

El punto único de cambio: data/store.js

TODOS los módulos llaman solo a Orbit.store — nunca tocan localStorage directo. Por eso migrar a backend es cambiar UN archivo, sin tocar los módulos.

// Orbit.store expone exactamente esta interfaz: all(coleccion) // → array de todos los registros get(coleccion, id) // → un registro where(coleccion, fn) // → filtrados insert(coleccion, obj) // crear update(coleccion, id, patch)// actualizar remove(coleccion, id) // borrar

Para conectar backend: reescribe esos 6 métodos para que lean/escriban en Firestore / Supabase / API REST en vez de localStorage. Los módulos no cambian.

Opción recomendada: Firebase (Firestore)

// 1. Crear proyecto en console.firebase.google.com // 2. Firestore Database → crear (modo producción) // 3. Cada "colección" de Orbit = una colección Firestore: // clientes, polizas, cobros, aseguradoras, gestiones, etc. // 4. En store.js, reemplazar la implementación: async function all(col){ const snap = await db.collection(tenant+'_'+col).get(); return snap.docs.map(d => ({id:d.id, ...d.data()})); }

Despliegue (hosting)

  • GitHub Pages — push al repo → Settings → Pages → main/root. Listo en 2 min. Ideal para demos.
  • Hosting propio + dominio — sube los archivos a tu servidor (cPanel/FTP/AWS S3). Configura HTTPS. Ideal para producción A&S: app.alianzasysoluciones.com.
💡 Multi-tenant: cada cliente tiene un prefijo en las colecciones (tenant_clientes) o una base de datos separada. Orbit.tenant en config.js define cuál se usa.

Seed (datos iniciales)

En data/seed.js está todo el dataset ficticio con versión __v. Para A&S: reemplaza las aseguradoras por las reales vinculadas, los asesores por el equipo real, y las metas. Sube __v para forzar recarga del seed.

⚠️ Datos ficticios en el core: Orbit 360 (comercializable) siempre lleva datos ficticios. Los datos REALES de A&S van solo en el proyecto A&S, nunca en el core.

4 · Integraciones

Se configuran desde Automatizaciones y Configuración. Sin claves, la plataforma funciona con heurística (sin costo); con claves, se activan las funciones reales.

🤖 IA — Gemini (económico)

1. aistudio.google.com → Get API key → copiar 2. Orbit → Automatizaciones → Asistente IA → pegar key → Activar 3. Activa: extracción real de PDFs (importadores), generación de cursos/copys, análisis crítico de Insights y Finanzas

⚡ Make (automatizaciones)

1. make.com → nuevo escenario → módulo Webhook → copiar URL 2. Orbit → Automatizaciones → Webhook Make → pegar → guardar 3. Probar disparo → verificar que llega el payload en Make 4. En Make: ramificar a WhatsApp Cloud / Gmail / Sheets / Canva

✉️ Correo (Outlook / Gmail / IMAP)

  • Outlook/M365: portal.azure.com → registrar app → permisos Mail.ReadWrite, Mail.Send, Calendars.Read → client_id + secret + tenant_id.
  • Gmail/Workspace: Google Cloud Console → OAuth 2.0 → scopes de Gmail API.
  • IMAP/POP3: para dominios propios — host, puerto, usuario, contraseña de aplicación. El correo del usuario es individual, no genérico.

📣 Redes y contenido

  • Metricool — programar/publicar en todas las redes desde un solo lugar.
  • Make → Canva → Metricool — flujo automatizado de generación + publicación.
  • Facebook, Instagram, LinkedIn, TikTok, YouTube — vía Metricool o Make.
  • HeyGen — videos de capacitación con avatar (Academia).
💡 Sin claves: todo funciona en modo demo/heurístico. Conecta las integraciones de A&S una a una conforme tengas las credenciales — la plataforma no se rompe si faltan.

Qué archivos exportar de tus herramientas actuales y dónde importarlos en Orbit. El importador inteligente mapea, cruza y complementa sin duplicar.

Excel/CSV de pólizas (con desglose de prima)Importación → Pólizas
PDFs de pólizas (carátulas)Importación inteligente (extrae datos) o expediente documental
Vehículos aseguradosImportación → Vehículos (se vinculan a pólizas auto)

⚠️ Documentos en Drive: NO migrar el histórico documental por espacio. El importador carga los documentos nuevos en la carpeta del cliente en Drive; cuando no encuentra la carpeta, la crea con el nombre completo del cliente y etiqueta los documentos.
💡 Histórico: al importar pólizas antiguas/canceladas NO se generan recibos pendientes (no alteran informes), pero SÍ se cuentan sus primas en analítica y comparativos.

6 · Actualizaciones post-migración

Cómo llevar las mejoras que hagamos en Orbit 360 (core) al proyecto A&S ya en producción — sin perder la personalización ni los datos reales.

El principio: core separado de configuración

Como A&S se personaliza solo en config.js y seed.js, puedes actualizar todo el resto del core sin tocar lo de A&S.

1
Mejora en Orbit 360 (este proyecto)
Seguimos mejorando aquí la versión comercializable. Cada versión sale como ZIP.
2
Copiar los archivos del core actualizados
En el proyecto A&S, reemplaza los archivos de core/, modules/ y styles/ con los nuevos. NO reemplaces config.js ni seed.js de A&S (tienen su personalización y datos reales).
3
Revisar config.js
Si la mejora añadió nuevas claves de configuración (un módulo nuevo, un rol nuevo), agrégalas manualmente al config.js de A&S — son pocas líneas.
4
Migrar el seed si cambió la estructura
Si una mejora cambió la forma de los datos (campos nuevos en pólizas, etc.), corre una migración puntual sobre los datos reales. El CHANGELOG indica cuándo aplica.
5
Verificar y desplegar
Prueba 5 registros end-to-end, luego push/deploy. Los datos de A&S (en backend) no se tocan en este proceso.

Flujo práctico con Codex / GitHub

# Si usas Codex con el ZIP: "Descomprime este ZIP en la raíz, conserva config.js y seed.js actuales, reemplaza core/ modules/ styles/ con los nuevos, y haz push a main." # Si trabajas directo en GitHub: git checkout -b update-vX.X # copia core/ modules/ styles/ del nuevo ZIP git add core modules styles && git commit -m "core vX.X" git push # revisa, mergea a main → deploy automático
💡 Regla simple: "el core viaja, la configuración se queda". Actualizas core/modules/styles desde Orbit 360; config.js y seed.js de A&S nunca se sobreescriben.

Con este documento tienes el camino completo: nuevo proyecto → backend → datos → integraciones → fuentes → actualizaciones. Orbit 360 sigue mejorando como producto; A&S es tu primer cliente real personalizado solo por configuración.
