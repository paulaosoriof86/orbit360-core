# Academia profunda — paquete Claude integral v1330

## Principio

Academia no debe ser biblioteca de textos. Debe ser un sistema de inducción, entrenamiento, adopción guiada y profesionalización por rol.

## Rutas a actualizar

### Cliente Portal

Debe aprender:

- reportar pago;
- adjuntar soporte;
- entender recibido/en revisión/aplicado;
- subir documentos;
- no confundir soporte con pago aplicado;
- revisar notificaciones y seguimiento.

### Cobros / Finanzas

Debe aprender:

- revisar soporte;
- validar sin aplicar;
- rechazar con motivo;
- solicitar aclaración;
- aplicar pago autorizado;
- conectar con M5 Conciliaciones;
- manejar país/moneda.

### Cliente360 / Asesor

Debe aprender:

- interpretar ficha 360;
- ver documentos por cliente/póliza;
- distinguir expediente aprobado vs revisión;
- revisar parches pendientes;
- crear gestiones sin cambiar datos sin autorización.

### Operativo / Gestiones

Debe aprender:

- recibir solicitudes del portal;
- relacionar documentos;
- solicitar información faltante;
- escalar bloqueos;
- mantener trazabilidad.

### Dirección / Admin

Debe aprender:

- aprobar/rechazar diffs;
- revisar auditoría;
- controlar roles/permisos;
- cambiar configuración con motivo;
- controlar riesgos de caja y documentos.

### IT / Seguridad

Debe aprender:

- tenant isolation;
- Storage futuro;
- credenciales seguras;
- integraciones pendientes;
- auditoría de acceso/lectura/descarga.

## Lecciones obligatorias nuevas

1. Reportar pago en Portal.
2. Revisar soporte en Cobros.
3. Validar reporte no aplicado.
4. Aplicar pago autorizado.
5. Conciliación M5: validada no aplicada.
6. Documentos y expediente.
7. Diffs desde documentos.
8. Cambio de roles/permisos con motivo.
9. Integraciones configuradas vs activas.
10. Storage pendiente vs conectado.

## Estructura de cada lección

Cada lección debe incluir:

- para qué sirve;
- por qué importa;
- quién la usa;
- qué datos lee;
- qué datos escribe;
- qué acciones/botones usa;
- qué cambia en otros módulos;
- qué notificaciones genera;
- qué errores evitar;
- caso práctico;
- evidencia de aprendizaje.

## Casos prácticos mínimos

1. Cliente reporta pago con soporte.
2. Cobros rechaza soporte por inconsistencia.
3. Cobros valida soporte pero no aplica pago.
4. M5 valida conciliación bancaria sin aplicar pago.
5. Cliente360 revisa documento de identidad con cambio propuesto.
6. Admin aprueba diff con motivo.
7. Superadmin cambia permisos sin dejar tenant sin administrador.
8. IT configura integración pendiente de conexión.

## Preguntas tipo quiz

Deben medir decisión, no memoria:

- ¿Qué estado debe ver un cliente al reportar pago?
- ¿Cuándo puede aplicarse un pago?
- ¿Qué pasa si falta moneda?
- ¿Puede un documento actualizar cliente automáticamente?
- ¿Qué acción requiere motivo?
- ¿Qué módulo debe usar el usuario para ver soporte de pago?
- ¿Qué texto no debe prometerse si Storage no está conectado?
